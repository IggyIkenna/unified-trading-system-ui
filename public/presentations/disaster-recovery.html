Circuit Breaker — Automated Recovery

The trading system implements a three-state circuit breaker that detects failures and progressively restricts operations to protect capital. No human intervention required.

Closed (Normal)
All systems operational. Trades execute normally. Continuous health monitoring active.
Half-Open (Degraded)
Anomaly detected. System sends probe requests. New orders throttled. Alerts fire to team.
Open (Protected)
Confirmed failure. All new orders halted. Existing positions held. Manual reset required.

What triggers the circuit breaker

  • Venue API failures or timeout spikes
  • Unexpected P&L drawdown beyond threshold
  • Execution quality degradation (slippage, missed fills)
  • Risk limit proximity or breach
  • DeFi transaction reverts or gas price spikes
  • Network connectivity loss to exchanges

What happens automatically

  • New order flow immediately halted
  • Existing positions preserved (no panic liquidation)
  • Alert sent to Slack, Telegram, and trading portal
  • Incident ticket auto-created in GitHub
  • AI debug agent begins root-cause analysis
  • State persisted — survives process restarts

Autonomous AI Debug & Recovery Agents

When something breaks, AI agents detect, diagnose, and propose fixes autonomously. Humans approve — agents execute.

PR Review Agent

Monitors all pull requests across ~60 repos. Detects bugs, regressions, type errors. If auto-fixable, opens a fix PR for human approval. The developer reviews the solution, not the bug.

Audit Agent

Periodic re-audit against a 14-section quality checklist. Grades the entire system. Findings auto-create GitHub issues with full context. Provides a board-level system health score.

Codex Sync Agent

Triggered on every code change. Auto-updates architecture documentation to reflect current state. Documentation is always current — never stale.

P&L Monitor Agent

Real-time P&L streaming across all strategies. Anomaly detection for unexpected drawdowns. Auto-escalates to humans. No analyst manually watching.

Design principle: agents suggest and flag — they never auto-execute trading decisions or deploy code without human approval. This is the right balance for FCA compliance and investor trust.

Version Control & Human Approval Gates

Breaking changes to the trading system require explicit human approval before they can reach production. No automated process can bypass this gate.

Major Version Bump Process

  • All repos start at 0.x.x — pre-1.0.0 safety net active
  • Breaking changes (feat!) on 0.x.x = MINOR bump only, never auto-crosses to 1.0.0
  • 1.0.0 graduation is a deliberate human decision
  • Post-1.0.0: any breaking change opens a GitHub approval issue
  • Telegram notifies the team with the approval link
  • Requires explicit /approve comment to execute

Two-Pass Quality Gates

  • Pass 1: Full suite — lint, tests, typecheck, security scan, codex compliance
  • Pass 2: Lightweight pre-merge — lint + format + typecheck + codex
  • Every repo has its own quality gate script
  • System Integration Tests validate cross-repo compatibility

Secrets & Credential Management

No one sees client credentials. Not even the trading systems at runtime.

GCP Secret Manager

  • All client API keys, wallet keys, and sensitive config stored in Secret Manager
  • Encrypted at rest (AES-256) and in transit (TLS 1.3)
  • Access controlled by IAM — only the specific service that needs a secret can read it
  • No human can view secret values via the console without audit trail
  • Secrets are injected at runtime via service account bindings — never written to disk
  • Version history maintained — rollback to any previous secret version

How Services Access Secrets

  • Services use Application Default Credentials (ADC) — no hardcoded auth
  • Each service has a dedicated service account with minimum required permissions
  • Interfaces are API-keyless by design — credentials injected by the calling service, never stored in the library
  • Secrets are fetched at startup, held in memory only, never logged
  • .env files contain no placeholder credentials — ADC is the default
  • Local development uses mock mode — no real credentials needed

Even if an attacker gained access to a running service's memory, they would only see credentials for that one service's scope. No service holds cross-system credentials. Lateral movement is architecturally impossible.

Key Rotation & Access Lifecycle

90-Day Rotation Policy

  • All API keys rotated every 90 days
  • Service account keys rotated on the same cycle
  • Exchange API credentials (Binance, OKX, etc.) rotated per policy
  • DeFi wallet keys: rotation via new wallet generation + fund migration
  • Every rotation is logged with timestamp, actor, and old/new key hash
  • Alerts fire if a key approaches expiry without rotation

Principle of Least Privilege

  • Every service account has only the permissions it needs
  • No service has admin-level cloud access
  • Read-only services cannot write; write services cannot delete
  • IAM policies reviewed quarterly

Employee Offboarding

  • Cloud IAM access revoked immediately on departure
  • GitHub organisation access removed
  • All service account keys associated with the individual rotated
  • VPN and SSH access terminated
  • OAuth sessions invalidated across all platforms
  • Offboarding checklist enforced — no step can be skipped
  • Post-offboarding audit: verify zero residual access within 24 hours

Access Reviews

Quarterly access reviews across GCP, AWS, GitHub, and all exchange accounts. Any permission not actively used is revoked. Default deny — access is granted only when justified and time-limited where possible.

Cloud Security Architecture

Google Cloud Platform (Primary)

  • IAM: Role-based access control with custom roles per service
  • VPC: Private network isolation — services not exposed to public internet
  • Secret Manager: All credentials encrypted and access-logged
  • Cloud Armor: DDoS protection and WAF rules
  • Audit Logs: Every API call, resource access, and admin action logged
  • Cloud Run / GKE: Container-level isolation, no shared compute
  • TLS 1.3: Encryption in transit for all internal and external communication

Amazon Web Services (Secondary)

  • IAM: Separate identity boundary, no cross-cloud credential sharing
  • VPC: Isolated network with security groups per service
  • KMS: Encryption keys managed independently from GCP
  • CloudTrail: Full audit trail of all AWS activity
  • S3: Bucket policies enforce encryption and access restrictions
  • Multi-cloud benefit: If GCP is compromised, AWS systems are unaffected — separate credentials, separate network, separate identity

No service mesh (deliberate). We use IAM + VPC + TLS at ingress instead of Istio/Envoy. Service mesh adds 2–5ms per hop — unacceptable for low-latency trading. Security is enforced at the infrastructure layer, not the application layer.
